# TiDB 数据库优化对刷题系统性能的影响

## 1. 为什么使用 TiDB 会比 Excel 文件更快？

### 1.1 数据访问方式对比

**Excel 文件访问：**
- 每次读取需要解析整个文件（即使只需要部分数据）
- 文件 I/O 瓶颈：大文件需要更多内存和CPU资源
- 并发访问困难：多用户同时访问时性能急剧下降
- 无索引支持：全表扫描，查询效率低

**TiDB 数据库访问：**
- 结构化存储：按行/列高效存储
- 索引加速：支持B+树索引、全文索引等
- 缓存机制：热数据自动缓存到内存
- 并发处理：支持高并发读写操作

### 1.2 具体性能提升

| 操作类型 | Excel 文件 | TiDB 数据库 | 性能提升 |
|---------|-----------|------------|----------|
| 按关键词搜索题目 | 全文扫描，秒级 | 索引查询，毫秒级 | 100-1000倍 |
| 按分类筛选题目 | 需要加载全部数据 | 索引查询，毫秒级 | 100-1000倍 |
| 多用户同时访问 | 串行访问，性能差 | 并发处理，性能稳定 | 10-100倍 |
| 大数据量查询 | 内存溢出风险 | 流式处理，稳定 | 显著提升 |

## 3. 针对刷题系统的具体优化建议

### 3.1 表结构优化

**索引设计：**
```sql
-- 问题内容全文索引（支持模糊搜索）
FULLTEXT INDEX ft_question (question)

-- 分类和来源索引（支持快速筛选）
INDEX idx_category (category)
INDEX idx_source (source)

-- 显示状态索引（支持条件查询）
INDEX idx_is_visible (is_visible)
```

**数据类型优化：**
- 使用TEXT类型存储长文本（问题、答案）
- 使用TINYINT存储布尔值（是否显示）
- 使用VARCHAR存储短文本（分类、来源）

### 3.2 查询优化

**分页查询：**
```sql
-- 高效分页（使用索引）
SELECT * FROM interview_questions 
WHERE category = '深度学习' 
ORDER BY id 
LIMIT 20 OFFSET 0;
```

**全文搜索：**
```sql
-- 全文搜索（支持中文分词）
SELECT * FROM interview_questions 
WHERE MATCH(question) AGAINST('神经网络' IN BOOLEAN MODE);
```

**组合查询：**
```sql
-- 多条件筛选
SELECT * FROM interview_questions 
WHERE category = 'NLP' 
  AND is_visible = 1 
  AND MATCH(question) AGAINST('分词' IN BOOLEAN MODE);
```

### 3.3 应用层优化

**连接池管理：**
```python
# 使用连接池避免频繁创建连接
from sqlalchemy import create_engine

engine = create_engine(
    "mysql+pymysql://user:password@host:4000/db",
    pool_size=10,
    max_overflow=20,
    pool_recycle=3600
)
```

**缓存策略：**
- 热门题目缓存到Redis
- 查询结果缓存
- 分页结果缓存

**批量操作：**
- 批量插入提高写入性能
- 批量查询减少网络往返
- 事务管理保证数据一致性

对于刷题系统来说，这些优化将直接提升用户体验，特别是在高并发访问和大数据量场景下，TiDB 能够提供稳定、快速的服务响应。
